# 全局工作约定

## Subagent 策略
- 除非任务明显属于单步、小范围、低收益的操作，否则优先判断是否可以拆分，并主动调用 subagent。
- 遇到以下场景时，默认优先委派 subagent：并行探索多个文件或模块、独立实现子任务、验证或回归检查、资料收集与对比。
- 当多个子任务彼此独立且并行处理可以缩短总耗时时，优先并行创建多个 subagent，而不是全部串行处理。
- 默认优先使用 `gpt-5.4` 模型，并将思考强度设置为 `xhigh`。
- 优先将只读探索工作交给 `explorer`，将具体实现工作交给 `worker`；如果存在更合适的窄职责自定义 agent，则优先匹配该 agent。
- 启动 subagent 前，先形成简短的高层计划，明确主 agent 当前立即推进的本地任务；不要把关键路径上必须立刻得到结果的阻塞工作盲目外包。
- 在 subagent 运行期间，主 agent 不要空等，应继续推进不重叠的本地工作；仅在 subagent 结果成为下一步阻塞时再等待。
- 委派任务必须具体、边界清晰、可交付，避免重复委派同一问题，也避免让多个 subagent 修改同一批文件。
- 对于简单、单步、几分钟内即可直接完成的任务，避免为了形式而调用 subagent。
- 最终答复前，需要汇总各 subagent 的产出、结论、风险与未决事项，并明确说明哪些工作由 subagent 完成。
- 如果用户明确要求不要调用 subagent，或当前环境限制不适合委派，则遵从用户要求或环境约束。
- 如果仓库级或子目录级 `AGENTS.md` 提供了更具体的 subagent 规则，则在对应作用域内优先遵循更具体的规则。

## Git 提交信息约束
- 仓库当前提交风格默认使用 `type(scope): 中文摘要`；若需要正文，标题与正文之间保留一个空行，正文使用 `- ` 项列出关键变更。
- 只要提交信息包含中文或其他非 ASCII 字符，禁止直接在 shell 命令参数里内联 message，例如 `git commit -m "中文..."`、`git commit --amend -m "中文..."` 或任何等价写法。
- 包含非 ASCII 字符的提交信息，必须先以 UTF-8 明确写入临时文件，或通过能精确控制字节内容的脚本构造后，再使用 `git commit -F <file>`、`git commit --amend -F <file>`、`git rebase --continue` 的编辑器文件、或等价的 UTF-8 安全方式提交。
- 不要仅凭终端显示判断提交信息是否正确。PowerShell、控制台编码、JSON 转义或中间脚本都可能把中文静默写成 `?` 或乱码。
- 每次创建、改写、cherry-pick、rebase、squash 或其他会生成新 commit 对象的操作后，只要该提交信息包含非 ASCII 字符，都必须做一次原始对象校验，再决定是否继续后续流程。
- 原始对象校验至少包含两步：
  1. 用 UTF-8 严格解码查看 `git cat-file commit <commit>` 的原始内容，确认标题和正文是预期中文。
  2. 检查标题和正文中是否出现异常占位或乱码，例如单独的 `?`、`锟斤拷`、`ç`、`æ`、`ä` 这类明显的 mojibake 片段；若出现则视为提交失败。
- 在 `git push` 之前，如果最近一次提交信息包含非 ASCII 字符，必须先完成上述校验；校验失败时，先修复 commit message，再允许推送。
- 若用户要求修复坏掉的 commit message，优先采用“保持 tree、parent、author 时间不变，只重写 commit message”的方式，避免把工作区未提交改动误带入历史。
- 若提交信息需要通过脚本生成，优先选择 UTF-8 严格模式；必要时用 Unicode 码点或十六进制字节构造 message，避免链路中任意一层自动转码。
- 若仓库未来新增 commitlint、commitizen、husky 或自定义提交脚本，以上编码与校验要求仍然优先，不能因为工具封装而跳过原始对象核验。
